# User Stories — Agentic Content Discovery, Phase 1

**Document ID**: stories_agentic_discovery_phase1.md
**Date**: 2026-04-04
**Status**: Draft
**Phase**: Phase 1 — Agentic Content Discovery
**Source BRD**: `agents/ba/brd_agentic_discovery_phase1.md` (BRD-007)
**Maintained by**: PM Agent

---

## Overview

These stories upgrade the discovery layer from keyword-driven Brave Search queries
to a system capable of surfacing genuinely curated writing — the kind found in
publications like The Browser, The Marginalian, or Arts & Letters Daily.

Four concrete improvements make up Phase 1:

1. **IndieWeb / Small Web source seeding**: A new source class that seeds from
   curated directories (Ooh.directory, Blogroll.org, IndieWeb wiki) and expands
   organically by following blogroll links found during crawls.

2. **Article body text extraction**: Before any LLM evaluation can happen, the
   system must fetch and extract the full body text of each candidate article.
   This is a prerequisite for story group 3.

3. **LLM-based content evaluation**: The title-specificity regex heuristic in
   `lib/discovery/qualityGate.ts` is replaced by a structured LLM scoring pass
   (Claude Haiku) that evaluates actual article content across five dimensions.

4. **Expanded search strategy**: Each topic carries a bank of five query
   formulations. Two queries execute per topic per run. A rotation cursor ensures
   the full bank is cycled over time. The bank is generated by LLM once at
   initialization and refreshed monthly.

All four groups build on the existing discovery infrastructure shipped in
Milestones 7 and 8. The RSS and NewsAPI fixed-source pipeline is not modified.
Downstream personalization, ranking, and feed API behavior are unchanged.

## Dependency Notes for the Architect

The four feature groups are not independent. The dependency order across groups is:

```
AGDISC-001 through AGDISC-004  (Small Web source seeding)
    — independent of body extraction and LLM eval; can run in parallel

AGDISC-005 through AGDISC-007  (Article body text extraction)
    — must ship before AGDISC-008 through AGDISC-010 (LLM evaluation)
    — AGDISC-005 BLOCKS AGDISC-008

AGDISC-008 through AGDISC-010  (LLM-based content evaluation)
    — depends on AGDISC-005 being shipped

AGDISC-011 through AGDISC-014  (Expanded search strategy)
    — depends on DISC-001 (existing topic configuration, already shipped)
    — independent of body extraction / LLM eval; can ship in parallel
```

Within the LLM evaluation group, body extraction (AGDISC-005 through AGDISC-007)
must be implemented and accepted before the LLM evaluator (AGDISC-008 through
AGDISC-010) is built.

## Architect-Level Decisions (Flagged — Do Not Resolve in Stories)

The following questions are implementation decisions the Architect must resolve
in the design document. They are called out explicitly in the relevant stories
so the Architect cannot miss them, but the PM does not constrain the choice.

1. **Small Web source state storage format**: The BRD specifies "a small JSON file
   or database table, consistent with the filesystem-first architecture." The
   Architect decides whether this is `data/small_web_sources.json` or a new DB
   table, and documents the choice. The PM's requirement is that the state
   (last-crawl timestamp, consecutive-zero-yield count, candidate source pool)
   persists across server restarts and is inspectable without a database client.

2. **Blogroll parsing scope**: The BRD specifies following blogroll links found
   during crawls. The Architect decides the specific HTML elements that constitute
   a "blogroll" (e.g., `<ul>` inside `<nav>`, OPML files, `rel="blogroll"` links,
   sidebar link lists), the maximum number of new sources added per crawl to
   prevent runaway expansion, and whether new sources are added immediately to the
   active pool or held in a staging queue for a first-pass check.

3. **LLM model selection**: The BRD specifies Claude Haiku (or cheapest available
   Anthropic model at time of implementation). The Architect confirms the specific
   model ID, documents the cost estimate per run at 30–60 candidates/day, and
   chooses whether to use streaming or synchronous completion for scoring calls.

4. **LLM prompt design and output parsing**: The Architect owns the exact prompt
   text, few-shot examples (if any), output format (structured JSON vs. parsed
   prose), and error handling for malformed model responses. The PM's requirement
   is that the output contains five numeric scores (1–5 each) and that composite
   averaging is deterministic and auditable.

5. **Query bank storage location**: The BRD specifies `data/query_banks.json` as
   an example. The Architect confirms this path and documents the JSON schema,
   including how per-topic rotation cursor state is stored (same file vs. separate
   state file vs. DB row).

6. **LLM query generation trigger**: The BRD specifies a one-time initialization
   step and monthly refresh. The Architect decides the exact trigger mechanism
   (a separate script, a CLI flag on the pipeline run command, an API endpoint
   guarded by `CRON_SECRET`, or a scheduled job entry) and documents how an
   operator runs it.

7. **Readability library integration**: The BRD specifies `@mozilla/readability`
   with `jsdom`. The Architect confirms this is the right choice given the
   server-side rendering environment (Next.js API routes), documents the package
   versions pinned, and notes any known incompatibilities with the current runtime.

### Group A — IndieWeb and Small Web Source Seeding

---

#### AGDISC-001 — Small Web Source State Store

**Priority**: P0
**Blocks**: AGDISC-002, AGDISC-003, AGDISC-004
**Depends on**: —

**As a** system that crawls Small Web sources on a schedule,
**I want** a persistent state store that tracks each known source's URL, last
crawl timestamp, and consecutive zero-yield count,
**so that** the crawler can enforce per-source cooldowns, deprioritize
low-yield sources, and expand the pool without redundant re-crawling.

#### Acceptance Criteria

1. A source state record exists for each known Small Web source. Each record
   contains at minimum: source URL, last-crawled timestamp (or null if never
   crawled), consecutive zero-yield count (integer, default 0), and active status
   (active or deprioritized).
2. State persists across server restarts. The storage mechanism (JSON file or DB
   table) is an Architect decision and must be documented in the design doc.
3. A source URL that already exists in the state store is not added again when the
   same URL is encountered during blogroll expansion (AGDISC-003). Deduplication
   is by normalized URL.
4. The state store is readable and inspectable by an operator without running a
   pipeline run — either by opening the JSON file or by a DB query, depending on
   the Architect's storage choice.
5. The initial state store is seeded with URLs from the three seed directories
   (Ooh.directory, Blogroll.org, IndieWeb wiki personal-site directory). The seed
   list is stored in code or a config file (not hardcoded inline in the crawler
   logic) so new seed URLs can be added without modifying crawler business logic.
6. State store writes are atomic: a write failure does not corrupt the existing
   state. (For JSON: write to a temp file, then rename. For DB: use a single
   upsert statement.)

#### AGDISC-002 — Small Web Source Crawl Scheduler

**Priority**: P0
**Blocks**: AGDISC-003, AGDISC-004
**Depends on**: AGDISC-001

**As a** system that fetches articles from Small Web sources,
**I want** the crawler to run once per day as part of the existing scheduled
pipeline, respecting per-source cooldowns,
**so that** the source pool is refreshed regularly without overwhelming small
personal sites with repeated requests.

#### Acceptance Criteria

1. The Small Web crawler runs once per day, triggered as part of (or immediately
   after) the existing `runPipeline()` call. The exact integration point within the
   pipeline execution order is an Architect decision and must be documented.
2. A source whose last-crawled timestamp is within the last 7 days is skipped
   entirely on that run. No HTTP request is issued to that source.
3. A source with an active status of "deprioritized" is crawled no more than once
   every 30 days, regardless of the 7-day rule. The effective cooldown for
   deprioritized sources is 30 days.
4. If a crawl of an individual source fails (HTTP error, timeout, parse error), the
   failure is logged with the source URL and error reason, and the crawler continues
   with the next source. A single source failure does not abort the crawl run.
5. Crawl run start and completion are logged with the count of sources attempted,
   sources skipped (cooldown), sources failed, and new articles yielded.
6. The crawler does not issue concurrent requests to the same domain. Per-source
   requests are sequential or throttled to avoid appearing as a DDoS to small sites.
   The Architect determines the specific throttle approach.

#### AGDISC-003 — Blogroll Discovery and Source Pool Expansion

**Priority**: P0
**Blocks**: —
**Depends on**: AGDISC-001, AGDISC-002

**As a** system that organically expands its source pool,
**I want** to parse blogroll links from each crawled site and add newly discovered
sites with valid feeds to the candidate source pool,
**so that** the Small Web source pool grows over time without requiring manual
curation after initial seeding.

#### Acceptance Criteria

1. When crawling a source page, the system parses the HTML for links that function
   as blogroll entries (links to other sites the author follows or recommends). The
   specific HTML elements and patterns that constitute a blogroll are an Architect
   decision and must be documented.
2. For each candidate URL discovered in a blogroll, the system checks whether the
   linked site has a valid RSS or Atom feed (by fetching a common feed discovery
   path or inspecting `<link rel="alternate">` tags in the site's `<head>`).
3. A newly discovered site that has a valid feed and is not already in the source
   state store is added to the store with last-crawled timestamp = null (eligible
   for crawling on the next run).
4. A newly discovered site that is already in the state store is not added again.
5. The maximum number of new sources added to the pool from a single crawl run is
   bounded. The specific cap is an Architect decision. This prevents runaway pool
   expansion from a single unusually large blogroll.
6. Blogroll discovery failures (no blogroll links found, feed check failed) are
   logged at debug level and do not affect the source's crawl record or active
   status.
7. Sites discovered via blogroll expansion are added to the source pool with the
   same data structure as seed-directory sources (AGDISC-001). There is no
   distinction between "seed" and "discovered" sources in the state store — all
   sources are treated equally by the crawler.

#### AGDISC-004 — Small Web Article Fetching and Feed Parsing

**Priority**: P0
**Blocks**: —
**Depends on**: AGDISC-001, AGDISC-002

**As a** system that ingests articles from Small Web sources,
**I want** to fetch each active source's RSS or Atom feed and extract article
candidates,
**so that** the articles can be passed to the body extraction step and quality
gate alongside Brave Search candidates.

#### Acceptance Criteria

1. For each source eligible for crawling on a given run (not in cooldown, not
   deprioritized-and-too-recent), the system fetches the source's RSS or Atom feed
   URL and parses it into a list of article candidates using the existing
   `rss-parser` dependency already in the project.
2. Each parsed article candidate is represented in the same shape as Brave Search
   candidates entering the quality gate: `title`, `url`, `description` (or excerpt),
   `publishedAt`, and `sourceName` (derived from the feed's channel title or domain
   name if the channel title is unavailable).
3. Only articles published within the discovery freshness window (the existing
   configurable constant established in DISC-004) are included as candidates.
   Older articles from the feed are discarded.
4. After a successful crawl, the source's last-crawled timestamp is updated in the
   state store. If the crawl produced zero qualifying article candidates (after
   freshness filtering), the source's consecutive-zero-yield count is incremented
   by 1.
5. If the consecutive-zero-yield count reaches 4 for a source, the source's active
   status is set to "deprioritized." It remains eligible for crawling on a 30-day
   interval (per AGDISC-002) but is not removed from the pool.
6. Article candidates sourced from Small Web feeds are passed to the same quality
   gate pipeline as Brave Search candidates (DISC-004 and its Phase 1 replacement,
   AGDISC-008). They are not held to a separate or more lenient standard.
7. The Small Web article candidates are deduplicated against the fixed-source
   pipeline and against each other using the same URL normalization already in
   place (DISC-005).

#### AGDISC-005 — Article Body Text Extraction Module

**Priority**: P0
**Blocks**: AGDISC-006, AGDISC-007, AGDISC-008
**Depends on**: —

**As a** discovery pipeline that evaluates article quality based on content,
**I want** a server-side module that fetches an article URL and extracts its
main body text,
**so that** the LLM evaluator has prose to assess rather than just a title and
one-sentence description.

#### Acceptance Criteria

1. A new module (location and name are an Architect decision) exposes a function
   that accepts an article URL and returns either extracted body text (a string) or
   a structured failure result with a reason code.
2. Extraction uses Mozilla Readability (`@mozilla/readability` package) with `jsdom`
   for DOM parsing, running server-side in Node.js. This is the same algorithm
   powering Firefox Reader Mode. The Architect confirms the specific package
   versions and documents any runtime constraints.
3. The function issues an HTTP GET to the article URL with a timeout of 8 seconds.
   If the request does not complete within 8 seconds, the function returns a failure
   result with reason `"fetch_timeout"`.
4. HTTP errors (any 4xx or 5xx response) result in a failure result with reason
   `"http_error"` and the status code recorded in the log.
5. If Readability returns null or an empty result (no extractable body), the function
   returns a failure result with reason `"extraction_failed"`.
6. If the extracted body text is fewer than 300 words, the function returns a failure
   result with reason `"below_minimum_length"`. Word count is by whitespace-separated
   token count.
7. On success, the function returns the extracted body text as a plain string. HTML
   tags are stripped. The text length is not bounded above (the LLM caller is
   responsible for any truncation before the API call).
8. The module is independently testable without invoking a real HTTP server. It can
   be tested by injecting mock fetch responses and mock Readability outputs.
9. No retry logic is implemented. A candidate that fails extraction is gone for that
   run. This is intentional per the BRD.

#### AGDISC-006 — Extraction Failure Logging and Pipeline Continuity

**Priority**: P0
**Blocks**: —
**Depends on**: AGDISC-005

**As a** system operator,
**I want** extraction failures to be logged with the specific failure reason and
for the pipeline to continue processing remaining candidates,
**so that** I can diagnose systemic extraction problems and individual failures
do not block the run.

#### Acceptance Criteria

1. Every extraction failure (all four reason codes from AGDISC-005: `fetch_timeout`,
   `http_error`, `extraction_failed`, `below_minimum_length`) is logged at debug
   level with: the article URL, the failure reason code, and (for `http_error`) the
   HTTP status code.
2. A failed extraction does not throw an exception that propagates to the discovery
   orchestrator. The article is silently dropped from the candidate pool for that run
   and the orchestrator continues with the next candidate.
3. Extraction failures do not count against the discovery quota. If the discovery run
   targets 6 qualifying articles and 4 candidates fail extraction, the remaining
   candidates are still evaluated and may contribute up to 6 articles. The quota is
   filled from surviving candidates, not reduced by extraction failures.
4. At the end of each discovery run, a summary log entry is written at info level
   with the total count of candidates attempted, candidates that failed extraction,
   candidates that failed LLM evaluation (once AGDISC-008 is shipped), and articles
   that qualified.
5. Extraction failures in the summary are broken down by reason code (e.g., "3
   fetch_timeout, 1 http_error, 2 below_minimum_length") so systemic issues are
   visible at a glance.

#### AGDISC-007 — Body Text Population on Article Record

**Priority**: P0
**Blocks**: —
**Depends on**: AGDISC-005

**As a** user reading a discovery article,
**I want** the article's body text to be available in the article detail view,
**so that** I can read the full article in-app when the discovery pipeline has
already fetched the content.

#### Acceptance Criteria

1. For every discovery article that passes the quality gate (post-AGDISC-008), the
   extracted body text from AGDISC-005 is stored in the `bodyText` field on the
   `Article` type (field already defined in `lib/types/article.ts`).
2. The `bodyText` field is persisted in the batch JSON file alongside other article
   fields. It is included in the `GET /api/articles/[id]` response (the field is
   already part of the Article type; no schema change is needed).
3. The `bodyText` field is not included in the `GET /api/feed/today` response (it
   would bloat the feed payload). This is already consistent with how the field is
   handled for any article type; the Architect confirms no change is required.
4. For fixed-pipeline (RSS/NewsAPI) articles that do not go through body text
   extraction, `bodyText` continues to be populated (or absent) exactly as it is
   today. This story does not change behavior for non-discovery articles.
5. If the article detail view already handles the presence or absence of `bodyText`
   gracefully (which it should given the field is optional), no UI change is
   required. The Architect confirms this is already true or flags the gap.

#### AGDISC-008 — LLM Content Evaluator Module

**Priority**: P0
**Blocks**: AGDISC-009, AGDISC-010
**Depends on**: AGDISC-005

**As a** discovery pipeline committed to surfacing genuine intellectual quality,
**I want** a module that sends extracted article body text to an LLM and receives
a structured quality score,
**so that** candidate evaluation is based on actual content, not just a
well-crafted title.

#### Acceptance Criteria

1. A new module (location and name are an Architect decision) exposes a function
   that accepts an article's title, description, and extracted body text, and
   returns either a structured score result or a failure result.
2. The function calls the Anthropic API using Claude Haiku (or the cheapest
   available Anthropic model confirmed by the Architect at implementation time).
   The model ID is a named constant, not hardcoded inline.
3. The function evaluates the article across exactly five dimensions:
   - Intellectual substance (1–5)
   - Originality (1–5)
   - Cross-disciplinary reach (1–5)
   - Evergreen durability (1–5)
   - Writing quality (1–5)
4. Each dimension score is an integer or single-decimal float in the range 1–5.
   The composite score is the arithmetic mean of all five dimension scores, rounded
   to two decimal places.
5. The prompt instructs the model to evaluate as a thoughtful, widely-read editor
   would — not as a classifier pattern-matching on surface signals. The exact prompt
   text and any few-shot examples are an Architect decision.
6. The function returns the five individual scores and the composite score in a
   structured result object. The output format (JSON from the model vs. parsed prose)
   is an Architect decision, but the result object's shape must be defined in a
   TypeScript type.
7. If the model returns a malformed or unparseable response, the function returns a
   failure result with reason `"parse_error"`. This does not throw or crash the
   pipeline.
8. If the Anthropic API call fails (network error, rate limit, timeout), the function
   returns a failure result with reason `"api_error"` and the error detail in the
   log. No retry logic in Phase 1.
9. The module is independently testable by injecting a mock API client. Tests must
   cover: a passing score, a failing score, a malformed response, and an API error.
10. The `ANTHROPIC_API_KEY` environment variable is required for this module. The
    `.env.example` file is updated to include this variable with a comment.

#### AGDISC-009 — Quality Gate: Replace Specificity Heuristic with LLM Evaluation

**Priority**: P0
**Blocks**: AGDISC-010
**Depends on**: AGDISC-005, AGDISC-008

**As a** discovery pipeline that filters candidates to a high quality bar,
**I want** the LLM content evaluation to replace the title specificity regex
heuristic in `lib/discovery/qualityGate.ts`,
**so that** quality decisions are based on actual article content, not
surface-level title patterns.

#### Acceptance Criteria

1. The title specificity heuristic (the regex-based scoring step currently in
   `lib/discovery/qualityGate.ts`) is removed. No other criterion in the quality
   gate is modified, removed, or reordered.
2. In its place, the quality gate calls the body text extraction module
   (AGDISC-005) followed by the LLM evaluator (AGDISC-008) as the final evaluation
   step for each candidate that has already passed the mechanical pre-filters
   (validator rules, freshness window, domain blocklist).
3. The execution order in the quality gate is:
   a. Validator rules (existing, unchanged)
   b. Freshness window (existing, unchanged)
   c. Domain blocklist (existing, unchanged)
   d. Body text extraction (new, AGDISC-005) — candidate is skipped if extraction
      fails
   e. LLM content evaluation (new, AGDISC-008) — candidate is scored
   f. Composite score threshold check — candidate passes if score >= 3.5
4. The LLM evaluation step is only reached by candidates that have already passed
   all three mechanical pre-filters. A candidate that fails the domain blocklist
   does not trigger an Anthropic API call.
5. The pass threshold (3.5 composite score) is a named configurable constant
   (e.g., `LLM_QUALITY_THRESHOLD`). Its location is consistent with the existing
   pipeline constants in `lib/config/feed.ts`. The Architect documents this
   constant in the design doc.
6. A candidate that fails the LLM threshold is logged at debug level with: the
   article URL, the five individual scores, the composite score, and the reason
   `"llm_threshold_not_met"`.
7. The quality gate module remains an isolated, independently testable pure function
   that can be unit-tested by injecting mock article objects and mock LLM responses.
   The addition of async I/O (the extraction and LLM calls) does not break the
   module's testability contract; the Architect designs the interface accordingly.
8. The existing quality gate unit tests are updated to reflect the removal of the
   specificity heuristic and the addition of the LLM evaluation step.

#### AGDISC-011 — Multi-Query Topic Bank Schema

**Priority**: P0
**Blocks**: AGDISC-012, AGDISC-013, AGDISC-014
**Depends on**: DISC-001 (existing topic configuration, already shipped)

**As a** discovery pipeline that issues diverse search queries,
**I want** each topic entry to carry a bank of five query formulations rather
than a single query string,
**so that** the system can vary its search angle across runs and surface a
broader range of sources.

#### Acceptance Criteria

1. The topic configuration schema (currently in `lib/discovery/topics.ts`) is
   extended to include a `queryBank` field: an ordered array of query strings.
   The existing single `query` field may be retained for backward compatibility
   or removed, at the Architect's discretion, as long as the execution logic reads
   from `queryBank`.
2. Each topic entry in the configuration carries exactly five query formulations in
   its bank at launch. The five formulations for each topic are authored by an LLM
   (AGDISC-014) but the schema must support them before the generation step runs.
3. The `queryBank` field is a required field in the `DiscoveryTopic` TypeScript type.
   A topic entry without a populated `queryBank` (empty array or missing field) is
   treated as invalid and the topic is skipped during a run with a warning logged.
4. The query bank file (`data/query_banks.json`, or the Architect's chosen path) is
   the authoritative source for query content at runtime. The static array in
   `lib/discovery/topics.ts` defines topic metadata (ID, label, weight); the query
   bank file defines query content. The Architect documents the relationship between
   these two artifacts and how runtime merging works.
5. The query bank is inspectable and manually editable by an operator (it is a plain
   JSON file, not embedded in compiled code). Manual edits to the query bank take
   effect on the next pipeline run without a code change or redeploy.
6. If the query bank file does not exist at startup (e.g., first deployment before
   initialization has been run), the pipeline logs a warning and falls back to the
   existing single-query behavior for affected topics. This fallback allows the
   system to run before the bank is populated.

#### AGDISC-012 — Query Rotation Cursor

**Priority**: P0
**Blocks**: AGDISC-013
**Depends on**: AGDISC-011

**As a** discovery pipeline that avoids repetitive querying,
**I want** a per-topic rotation cursor that tracks which query in the bank was
last executed,
**so that** the full query bank is cycled through over time rather than always
running the same one or two queries.

#### Acceptance Criteria

1. For each topic, a rotation cursor records the index of the last query executed
   from that topic's bank. On each discovery run, the system selects the next two
   queries in sequence starting from `(last_cursor + 1) mod bank_size`.
2. After executing two queries for a topic, the cursor is updated to reflect the
   index of the last query executed in that run.
3. When the cursor reaches the end of the bank, it wraps to the beginning (circular
   rotation). After executing queries at indices 3 and 4 from a 5-query bank, the
   next run executes indices 0 and 1.
4. Rotation cursor state is persisted between runs and survives server restarts. The
   storage mechanism (same file as the query bank, a separate state file, or a DB
   table) is an Architect decision and must be documented. The PM's requirement is
   that the state does not reset on server restart.
5. When a topic's query bank is refreshed (AGDISC-014), the rotation cursor for that
   topic resets to 0. A bank refresh always starts the new bank from the first query
   on the following run.
6. If a topic has fewer than 2 queries in its bank (e.g., manually edited down to
   1), the system executes all available queries for that topic rather than
   failing. A warning is logged.
7. Cursor state for all topics is written atomically. A write failure during a run
   does not corrupt the state for topics whose cursors were already updated in
   that run.

#### AGDISC-013 — Two-Queries-Per-Topic Execution

**Priority**: P0
**Blocks**: —
**Depends on**: AGDISC-011, AGDISC-012

**As a** discovery pipeline that searches across diverse query angles,
**I want** the search execution step to issue exactly two queries per topic per
run (selected by the rotation cursor),
**so that** each run explores a different slice of the topic's query bank and
the overall candidate pool is more varied than single-query execution.

#### Acceptance Criteria

1. On each discovery run, for each active topic, exactly two queries are issued to
   the Brave Search API. The specific queries are the two entries selected by the
   rotation cursor (AGDISC-012).
2. Results from both queries for a topic are combined before quality gate evaluation.
   If both queries return the same URL, only one instance is retained (deduplication
   within the topic's result set).
3. The total number of Brave Search API calls per run is (number of active topics)
   × 2. For the current 12-topic configuration, this is 24 calls per run. At a
   once-daily cadence, this is approximately 720 calls per month — within the Brave
   Search free tier of 2,000 calls per month.
4. If one of the two queries for a topic returns an error or empty results, the other
   query's results are still used. A single-query failure within a topic does not
   discard the topic's contribution entirely.
5. The per-topic candidate pool size configurable constant (number of results
   requested per query) is unchanged from the existing implementation. Each query
   continues to request the same number of results it did before.
6. The existing topic weight feedback loop (DISC-009) continues to operate
   correctly. The topic weight applies to how often (or with what priority) a topic
   is included in a run, not to which queries within that topic are selected. The
   cursor selects queries within a topic; weight governs topic-level sampling.

#### AGDISC-014 — LLM-Generated Query Bank Initialization and Refresh

**Priority**: P0
**Blocks**: —
**Depends on**: AGDISC-011

**As a** system operator setting up Phase 1 for the first time,
**I want** a one-time initialization step that uses an LLM to generate five
query formulations per topic and write them to the query bank,
**so that** the query bank is populated with high-signal, curatorial search
queries without requiring manual authoring for every topic.

#### Acceptance Criteria

1. A runnable initialization script or API endpoint (behind `CRON_SECRET`) generates
   query bank content for all configured topics by calling the LLM. The trigger
   mechanism is an Architect decision and must be documented in the deployment
   notes for this phase.
2. The generation prompt instructs the LLM to write queries the way a master
   curator would search: not generic keyword phrases, but precise formulations that
   surface high-signal, niche, cross-disciplinary writing. Example character: for
   "materials science," an acceptable generated query is "overlooked properties of
   materials that changed how something was made" rather than "materials science
   articles." The exact prompt is an Architect decision.
3. The LLM generates exactly five query strings per topic. If the model returns
   fewer than five for a topic, the initialization step logs a warning and pads the
   bank with available queries (rather than failing the entire initialization).
4. The generated query bank is written to the configured file path (e.g.,
   `data/query_banks.json`). The file is human-readable JSON and can be inspected
   and manually edited after generation.
5. Running initialization a second time (re-initialization) overwrites the existing
   query bank for all topics and resets all rotation cursors to 0. A log entry is
   written recording the re-initialization event and timestamp.
6. A monthly refresh mechanism generates a fresh query bank (same prompt, same
   output structure) and replaces the existing file. The Architect decides whether
   this is a separate scheduled job, a flag on the existing initialization trigger,
   or a manual operator action, and documents the choice.
7. The LLM used for query generation is the same model as the content evaluator
   (AGDISC-008), for cost and configuration consistency. The Architect may override
   this if a different model is more appropriate for the generation task.
8. The `ANTHROPIC_API_KEY` environment variable (added in AGDISC-008) is the only
   new environment variable required for query generation.

## Definition of Done (Phase 1)

All P0 stories are accepted when:

1. The Small Web crawler runs once per day and adds article candidates from
   IndieWeb / personal-blog sources to the discovery pipeline alongside Brave
   Search candidates.
2. The source pool expands organically over time: crawling a known source page
   discovers blogroll links, and newly found sites with valid feeds are added to
   the state store for future crawling.
3. Sources in the pool are not re-crawled more often than once every 7 days.
   Sources that have produced zero qualifying articles across 4 consecutive crawls
   are deprioritized to a 30-day interval.
4. Every discovery candidate (from any source class) that passes the mechanical
   pre-filters (validator rules, freshness, domain blocklist) is sent through body
   text extraction before the quality gate makes a final pass/fail decision.
5. Extraction failures (timeout, HTTP error, too-short text, unparseable body) are
   handled without aborting the pipeline run. The candidate is dropped; the run
   continues.
6. The title specificity regex heuristic in `lib/discovery/qualityGate.ts` is
   removed. The quality gate's final step is LLM composite scoring.
7. An article must score 3.5 or higher (composite average of five 1–5 dimensions)
   to pass the quality gate. The threshold is a named configurable constant.
8. Articles that pass the full quality gate (including LLM scoring) have their
   extracted body text stored in the `bodyText` field on the article record.
9. Each topic's query bank contains five query formulations. Two queries per topic
   are executed per run, selected by a rotation cursor that cycles through the bank
   over time.
10. The rotation cursor persists across server restarts. Re-running the pipeline
    on the same day does not reset the cursor.
11. The query bank can be refreshed (monthly or on demand) by running the
    initialization script. Refreshing resets rotation cursors to 0.
12. The `GET /api/feed/today` response shape is unchanged. All existing integration
    tests pass without modification.
13. The existing RSS and NewsAPI fixed-source pipeline behavior is unchanged.

The P1 story (AGDISC-010 — LLM Evaluation Cost and Run-Time Observability) is
accepted when:

14. Each pipeline run log entry includes the count of LLM evaluation calls, pass
    rate, and total wall-clock time spent on LLM calls.

## Notes for the Architect

- **Integration point for Small Web candidates**: Small Web article candidates
  (from AGDISC-004) should enter the quality gate at the same point as Brave Search
  candidates. The discovery orchestrator in `lib/discovery/run.ts` is the right
  place to merge candidate pools before quality gate evaluation. The Architect
  decides the exact merge point and documents it.

- **Quality gate is currently synchronous**: The existing `qualityGate.ts` is a
  pure synchronous function. Adding body text extraction (HTTP fetch) and LLM
  evaluation (API call) makes it async. The Architect must update the function
  signature and all callers. This is a required interface change; flag it clearly
  in the design doc.

- **`ANTHROPIC_API_KEY` is a new required environment variable**: Document it in
  `.env.example` with a comment explaining it is required for both content
  evaluation (AGDISC-008) and query bank generation (AGDISC-014). Also document
  it in ARCHITECTURE.md alongside `BRAVE_SEARCH_API_KEY`.

- **Body text truncation before LLM call**: Readability may return very long body
  text. The Architect must decide the maximum token budget to send to Claude Haiku
  per evaluation call (e.g., first 2,000 words) and document the truncation
  strategy. This is a cost and latency decision, not a PM decision.

- **Quota constraint unchanged**: `DISCOVERY_ARTICLES_PER_DAY = 6` is unchanged.
  The expanded candidate pool (Brave Search + Small Web feeds) feeds into the same
  quota enforcement (DISC-006). The quota is not increased in Phase 1.

- **`data/` directory items added in Phase 1**: At minimum, `data/query_banks.json`
  and a Small Web source state file are new entries in `data/`. Update
  `.gitignore` documentation accordingly (these files should be git-ignored in
  production, similar to `data/batches/`).

- **Backward compatibility with DISC-009 (topic weight feedback loop)**: The
  expanded search strategy changes which queries run but does not change which topic
  an article is attributed to. The `discoveryTopic` field continues to record the
  topic label, not the specific query used. The topic weight feedback loop remains
  unaffected.
